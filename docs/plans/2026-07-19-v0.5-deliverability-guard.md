# v0.5 Deliverability Guard — corrected plan

Supersedes the wave-1 A–E patch design. Folds the adversarial verify, the Codex ultra review (verdicts binding; its probe run and file:line anchors spot-checked against the v0.4.5-dev tree), the provenance-ranked practitioner corpus, and three arbitration rulings resolved by primary-source fetch (2026-07-18). Every number below traces to a named source; practitioner figures are labeled as practitioner experience, never provider policy.

## 1. Context

**Incident.** An agent sent ~500 emails over ~2 days (hypothesis — reconstruct by hour from Resend logs) through native Resend from `letter.sharpestmove.com`, a deliberately created but zero-history subdomain, with **no webhooks configured** — no bounce/complaint/open visibility at all. Result: spam-foldering across providers. Not this CLI's send, but exactly the failure class this CLI exists to close.

**Thesis.** Resend enforces nothing pre-send: day-0 sending is allowed the moment a domain verifies, the API permits ~1,000 emails/second (10 req/s x batch 100 — resend.com/docs/api-reference/rate-limit) against Resend's own day-1 warm-up guidance of 150/day (docs/knowledge-base/warming-up), and the 0.08% complaint / 4% bounce lines are reactive account-pause thresholds (AUP: "account may be shutdown without warning" — resend.com/legal/acceptable-use), not preventive throttles. Managed platforms carry the missing layer — Buttondown runs 62 pre-flight checks + mandatory double opt-in + human import review (buttondown.com/blog/deliverability), Mailchimp Omnivore hard-blocks pre-send on never-contacted addresses with no user override (mailchimp.com/help/about-omnivore), Postmark human-approves accounts and queues on pause — but none of them can be scripted by an agent. mailing-list-cli closes the gap client-side. Notably, Resend itself treats enforced warm-up as non-negotiable at the IP layer (managed dedicated IPs auto-warm with no documented disable — docs/knowledge-base/how-do-dedicated-ips-work); the recipient-level ramp is the genuinely missing guard.

**Honest enforcement framing (Codex-required — do not oversell).** The guard is **best-effort protection for cooperative agents using this database**. A filesystem-capable agent can bypass any local guard:

- pass `--override-ramp` (if it exists and is discoverable)
- set `guards.ramp_enabled = false` in config.toml
- repoint `MLC_CONFIG_PATH` / `MLC_DB_PATH` at a fresh config/ledger (src/paths.rs:3, src/paths.rs:13)
- edit the SQLite DB directly
- loop `broadcast preview`, which sends without the shared preflight (src/broadcast/pipeline.rs:690, :783)
- invoke `email-cli` or the native Resend API directly with the key
- run from a second machine / second DB the ledger never sees

The real boundary is **withholding the Resend key from the agent** (out-of-band override token: open question §6). Document this in agent-info and the embedded skill; never claim "agent burns become impossible."

## 2. Immediate v0.4.6 P0 fixes — ship BEFORE any guard work

Live bugs in v0.4.5, all Codex-found, spot-checked in the working tree.

**(a) Resume re-sends delivered recipients — live P0.** Pipeline writes recipient `status='sent'` (pipeline.rs:543-546); webhook/poll dispatch then mutates it to `delivered`/`bounced`/`complained` (src/webhook/dispatch.rs:36-37 for delivered; :42/:73 arms). The resume skip-set selects `status='sent'` only (src/db/mod.rs:1833, `broadcast_recipient_already_sent_ids`). Scenario: send 300/500 → poll runs → 300 flip to `delivered` → crash → `broadcast resume` re-sends all 300. Fix now: widen the skip-set to `IN ('sent','delivered','bounced','complained')`; durable fix (immutable `accepted_at`) lands in step 2.

**(b) Batch response non-validation.** `batch_send` accepts any returned array length and maps missing IDs to `""` (src/email_cli.rs:501, :515); the pipeline silently skips unmatched/empty pairs without incrementing `failed_count` (pipeline.rs:511), then can mark the broadcast `sent`. Resend's permissive partial batch validation makes ignored per-entry `errors[]` a live hazard (resend.com/changelog/batch-validation-modes). Fix: exact cardinality, non-empty unique IDs, recipient correlation, parse `errors[]`; mismatch fails the chunk.

**(c) No idempotency key — upstream email-cli prerequisite.** Batch send passes only `--file` (email_cli.rs:439-446) and retries transient failures up to 4x (`MAX_RETRIES=4`, :27; loop :453-467). A timeout after Resend accepts but before email-cli returns duplicates a 100-recipient chunk on retry. Resend supports 24-hour batch idempotency for exactly this (resend.com/docs/dashboard/emails/idempotency-keys). Fix: email-cli grows idempotency-key support + strict validation mode + capability reporting; mailing-list-cli persists a deterministic identity/broadcast/chunk key before sending, reuses it on every retry, records the 24h expiry, refuses blind retry after expiry. (Correction: `AppError::RateLimited` is NOT unused — batch retries already return it, email_cli.rs:472; its suggestion emits a literal `<id>` placeholder, :479-480 — fix alongside.)

**(d) Partial-send double-envelope + unresumable states.** Partial outcome: success envelope printed, then `Err` returned (src/commands/broadcast.rs:177-186) → main.rs:50 prints a second envelope. Any partial becomes broadcast `status='sent'` — only total failure becomes `failed` (pipeline.rs:580-584) — and a canonical snapshot is written (:588); resume accepts neither `sent` nor `failed` (pipeline.rs:75); the error nonetheless instructs the agent to resume (broadcast.rs:195-197). Fix: resumable `partial` status; single outcome envelope via an outcome type carrying status + exit code; exit 1; structured per-chunk failures; no final `sent_at`/snapshot until all intended recipients are accepted or intentionally suppressed. (Wave-1's Patch D anchored this at pipeline.rs:364 — wrong: that is the WAL indeterminate/manual-SQL path (:347); partial handling lives in the command wrapper.)

**(e) Preflight lock leak + 30-min lock-break hazard.** Lock is acquired before preflight (pipeline.rs:105, preflight at :161) and many early error paths leave it held (acknowledged at :86) — any new preflight refusal (ramp!) would strand the broadcast in `sending`. Static 30-min staleness (pipeline.rs:112, db/mod.rs:1345) lets a second process break a legitimate >30-min send. Fix: RAII/guard cleanup releasing the lock on all error paths; renewable lease/heartbeat with PID+host ownership.

**(f) Poll blindness.** `poll_events` requests one 100-item page, ignores `has_more`, advances the cursor to the last item id (src/webhook/poll.rs:23, :110), reads only `last_event`, and supplies no bounce subtype (:100 — "poll path doesn't expose bounce subtype"), so the dispatcher never classifies poll-sourced bounces as permanent or auto-suppresses them (dispatch.rs:42 arm). Later state changes on already-seen email IDs are missed — the repo admits this (db/mod.rs:2089). Fix: paginate on `has_more`; bounded recent-window rescan/diff for state changes; carry bounce subtype through poll → dispatch so suppression fires; record provider occurrence time (event windows currently use local ingestion time, db/mod.rs:1696).

## 3. Corrected patch sequence — 9 steps (replaces wave-1 A–E)

Wave-1's A→B→C→D→E is not independently shippable: A strands locks and races allowance, B reads telemetry that cannot see delayed outcomes, E (safe smoke) was last but is needed first. Codex's order is binding. Smoke strategy throughout: **Resend simulation addresses, never real recipients** (`delivered@` / `bounced@` / `complained@` / `suppressed@resend.dev` — resend.com/docs/dashboard/emails/send-test-emails); conformance/config patches need no external sends at all.

**Step 1 — upstream email-cli transport prerequisite.** Idempotency-key flag; explicit strict validation mode; response cardinality / non-empty-ID / `errors[]` parsing; capability+version reporting so mailing-list-cli can refuse to run against a pre-idempotency email-cli. Smoke: batch to simulation addresses, kill mid-flight, retry, assert exactly-once in the Resend dashboard.

**Step 2 — send-correctness foundation (mailing-list-cli).** The P0 fixes (a)(d)(e) done properly:
- **Immutable `accepted_at`/`accepted_count` ledger**, separate from mutable delivery status; all usage accounting and resume decisions read immutable accepted state + outstanding reservations, never `status='sent'` counts (which self-erase as telemetry arrives).
- **Frozen send plan**: freeze identity, template subject/HTML/text bytes, recipient membership, order, chunking, compliance material, and idempotency keys before the first external call. Today the list/segment re-resolves per run (pipeline.rs:793), unsubscribe tokens embed the current run timestamp (:389), the request hash includes rendered bytes (:481), and the snapshot lands only after completion (:588) — so crash+resume can send different content to different recipients under one broadcast. Stable contact sorting alone does not create a stable request.
- Partial/resume state machine + single outcome envelope (P0 d); RAII lock + heartbeat (P0 e); per-chunk suppression recheck (suppression currently loaded once before all chunks, pipeline.rs:203); run/attempt/chunk IDs as Resend tags (only `broadcast_id` today, :477) for dashboard reconciliation.

**Step 3 — framework/agent contract foundation.** `Cli::try_parse` entry rewrite, JSON help/errors, `--quiet` + `Ctx{format,quiet}`, canonical manifest shape, doctor/config contract, **pinned conformance probe in CI and release** (both currently stop at ordinary Rust checks — .github/workflows/ci.yml:36, release.yml:70). Detail in §5.

**Step 4 — isolated simulation.** `broadcast send <id> --simulate` against a **dedicated simulation namespace/table** — not rewritten production recipients. Wave-1's "rewrite recipients, no DB sent-marks" is contradictory: the pipeline takes the real lock, inserts real recipient rows and WAL attempts before calling Resend (pipeline.rs:105, :194, :484), so you'd either leave `prepared` poison-pills, contaminate real contact state, or store nothing and blind the breaker. Send **one instance per official outcome address** including `suppressed@resend.dev`; no production contact or ramp mutation; ideally a dedicated Resend test project. Refuse only reserved/known-invalid fabricated domains (the CLI cannot know `test@real-company.com` is fake — invented addresses hard-bounce against the account, per the send-test-emails doc); present `--simulate` as the safe path. This becomes the standard pre-tag smoke. Simulation validates API/render/event plumbing — it is not placement proof.

**Step 5 — identity-scoped ramp governor.** The core guard, redesigned:
- **Atomic reservations**: reserve allowance in the same `BEGIN IMMEDIATE` transaction before any external call. The send lock serializes only one broadcast id (db/mod.rs:1314) — two broadcasts can both read the same remaining allowance and both send. Reservations scoped to a defined sending identity; include `prepared`/in-flight/acked attempts; release only capacity proven not accepted. Ramp clock starts on **first reservation/acceptance**, not "first successful send" (misses partially/externally accepted sends).
- **Rolling 24h/60min windows or token bucket** — no UTC calendar buckets (day-1 capacity at 23:59 + day-2 at 00:00 = burst). Hourly smoothing from day 1: derive `hourly_cap = max(10, ceil(daily_cap/10))` (token-bucket refill). This closes both Codex holes: Resend's table has no hourly cap days 1–3 (150 in seconds = the original failure mode) and carrying day-7's 150/hr to day 42 then dropping it is a cliff — a derived cap scales with the daily allowance and never cliffs. Derivation is **our choice, flagged tunable**, not Resend's number.
- **Checked `u64` math** with explicit upper bounds — the existing-domain formula (10,000 x 1.5^(N-7)) exceeds `u32` on day 39.
- **Govern the frozen pending plan**, not the original list: preflight currently evaluates the full resolved target pre-suppression/pre-resume-filter (pipeline.rs:157, :194, :252), so a resume with 20 pending would be refused because the original list held 500.
- **`unknown` onboarding profile**: config absence silently yields defaults (src/config.rs:138). A v0.4 upgrader with established history must NOT wake up capped at 150/day. Upgraded DBs get `profile='unknown'` → governor warns on every send + health flags it, does not cap; `ramp init --profile <new_domain|existing_domain|recovery> --confirm` is the one-time human choice, printing the resulting caps before enabling enforcement; fresh v0.5 DBs require it before first send. `ramp_profile` is a validated enum, not a string. Ship migration notes + a documented rollback with the v0.5 upgrade (unset profile / disable enforcement returns to v0.4 behavior).
- **Guard every external sending path including `broadcast preview`** (pipeline.rs:690, :783).
- **Refusal envelope**: exit 4 (`ramp_daily_cap_exceeded` / `ramp_hourly_cap_exceeded`) with machine-readable `retry_at`, `allowed_recipients`, `requires_human_authorization` — prose-only suggestions invite retry loops and placeholder copying. **The first refusal does not advertise any override flag**; it says a smaller send is allowed now, a larger one after `retry_at`, and that raising caps requires explicit human authorization (`ramp init`/config). Burn-cost copy: "weeks to months of reputation recovery; no provider publishes a clock" (the wave-1 "4–12 weeks" figure was internally inconsistent and is refuted).
- **Dry-run**: exit 0 with `would_block`, allowance, `retry_at` — a report, not a RateLimited failure.
- Stable cohort/split command so "send to at most N today" is a tested, followable suggestion.
- Numbers: §4 table. Smoke: simulation namespace + synthetic ledger rows; assert refusal envelopes and reservation atomicity under two concurrent sends.

**Step 6 — telemetry correctness.** Prerequisite for any breaker: signed webhook ingestion (`svix-id` dedup, at-least-once, out-of-order tolerance, provider occurrence timestamps, coverage/backlog metadata — resend.com/docs/webhooks/introduction, /event-types) or a bounded recent-window rescan/diff atop fixed polling (P0 f). Parse SMTP reputation strings as the live alarm: `550 5.7.1` "low reputation of the sending domain", `421 4.7.0` temporary variants, and Gmail's per-identity `421-4.7.28` DKIM/SPF-domain rate limiting (observed throttling new DKIM domains at ~1,000–2,000/hr — Stefano Bagnara, mailop msg21216/msg20931; Brandon Long on reputation by spfdomain:dkimdomain pair and historical-usage throttling, mailop msg23242). Bucket outcomes by recipient MX for per-provider gates. Postmaster arbitration (2026-07): **legacy PMT v1 is alive — deprecation postponed** (support.google.com/mail/answer/16594218); the Domain/IP Reputation dashboards *will be* retired and are absent from v2, so treat legacy Domain Reputation as bonus diagnostics while it exists (low-volume "no data" expected — privacy floor, support.google.com/mail/answer/9981691) and **gate machine logic on v2 Spam Rate + Compliance Status only**.

**Step 7 — persistent circuit breaker.** Identity/account-wide persistent policy stop, not a per-run check: trips on any complaint while delivered < 1,250 (0.08% = 1/1,250 — Resend AUP), bounce ≥ 4%, or SMTP reputation-string detection; requires a **delayed-event soak period** between cohorts (immediate post-chunk polling observes `sent`/`queued`, not complaints — that was wave-1 Patch B's blindness). **Exit 3 with explicit human reset** (`ramp breaker reset --confirm`), NOT exit 4 — "wait, then retry" invites an LLM to resume the harmful send. Machine-readable state; breaker state survives process restart and blocks all send paths.

**Step 8 — health/doctor checks.** Tri-state `pass`/`fail`/`unknown` — incomplete telemetry must never read as healthy (`historical_send_rates` reports zero when delivered=0 even with failures, and divides bounce/complaint by delivered — db/mod.rs:2114, :2135). DMARC: resolve the **effective** policy — a missing `_dmarc.<From-domain>` is not failure, the org domain's `p=`/`sp=` inherits (and root `p=reject` with no `sp=` silently puts the subdomain at reject); **PSL-based root-domain detection**, not dot counting; `dig` with timeout, multi-string TXT, multi-record detection, `unknown` when unavailable. Fix the friendly-From parser's trailing `>` (health.rs:102) and `Some("")`-as-valid-address (health.rs:85). Poll cursor is an email id, not a freshness timestamp (poll.rs:14) — don't fake freshness from it. Check MX/A on the From domain (a reply-inviting recovery strategy is dead if replies bounce) and lint `noreply@` localparts. Enforce the documented 16-byte unsubscribe-secret minimum (unsubscribe.rs:28 accepts anything). Doctor semantics: §5.

**Step 9 — engagement ordering: only if signals improve, else omit.** The proposed `opens + clicks*3` score is structurally weak: Apple MPP proxy-fires >55% of opens regardless of reading (litmus.com/email-client-market-share), enterprise scanners auto-fetch links (Microsoft Safe Links docs; Atkins on Barracuda clicking all links), events are deduped to one row per email/type (migrations.rs:191) and the poll path sees only last state. Ship engagement-first ordering only when replies, recorded reconsent, or first-party activity are available as signals; opens serve as a negative ceiling only ("no open in N months"). Never auto-suppress on a bare one-click-unsubscribe GET — RFC 8058 mandates POST precisely because scanners fire GETs.

## 4. Ramp defaults table

The governor enforces **practitioner floor as default, Resend ceiling as hard max**. `new_domain`/`recovery` profiles run the floor; `existing_domain` may run up to the ceiling; nothing ever exceeds the ceiling. Floors are practitioner experience (labeled as such in all CLI copy), the ceiling is Resend's published schedule.

| Day | Resend ceiling (new domain)¹ | Practitioner floor (default)² |
|---|---|---|
| 1 | 150 | 25 |
| 2 | 250 | 25 |
| 3 | 400 | 40 |
| 4 | 700 (50/hr) | 40 |
| 5 | 1,000 (75/hr) | 60 |
| 6 | 1,500 (100/hr) | 60 |
| 7 | 2,000 (150/hr) | 90 |
| 8+ | x1.4/day, cap day 42 | x1.5 per 2–3 clean days |

¹ resend.com/docs/knowledge-base/warming-up — re-confirmed digit-for-digit 2026-07-18. Existing-domain ceiling: 1,000/2,500/5,000/5,000/7,500/7,500/10,000 daily, 100–2,000 hourly, x1.5/day.
² Floor rationale, cited: Bento starts week 1 at **10–100/day** and observes Gmail tolerating ~10–20 per few hours (Jesse Hanley, bentonow.com/posts/domain-warmup-guide — Bento's observation, not Google policy); Customer.io stage 1 = **100/day**, growth ≤ **1.5x** per stage — "doubling volume at each step is too aggressive" (docs.customer.io/journeys/channels/email/deliverability/domain-warming); Klaviyo gates advancement on **≥20% opens** and auto-trims unengaged with a flagged override (help.klaviyo.com/hc/en-us/articles/20413890435355, /4402636609691) — we gate on complaint/bounce/reply instead, opens being MPP-polluted; SES sandbox = **200/24h** with quota growth earned by clean sending (docs.aws.amazon.com/ses/latest/dg/manage-sending-quotas.html). Hourly smoothing from day 1 = ceil(daily/10), ours, tunable (§3 step 5).

**Enforced-warmup prior art** (wave-2, corrected): default-on managed warm-up = **SES** (45-day percentage plan, default-on but account-disableable; managed tier per-ISP adaptive with decay — dedicated-ip-warming.html, managed-dedicated-sending.html), **Bird** (~30-day auto with shared-pool overflow — bird.com docs), **Resend itself** (managed dedicated IP pools, no documented disable). Opt-in enforced: SendGrid (42-day hourly caps, 72h overflow retry), Mailgun (stage caps, overflow to shared). Hard pre-send block: Mailchimp Omnivore (never-contacted trigger, no self-serve override). Human gate + queue-on-pause + forced stream separation: Postmark. The original "SES is the only default-on" claim is refuted — the corrected spectrum above is binding. Industry pattern to copy: meter don't block, degrade gracefully (reroute → throttle → queue), overrides are explicit, logged, responsibility-shifting human acts. Deliberately not built, ever: fake-engagement warm-up network integration — Google revoked Gmail API access for warmup-inbox tools as policy abuse, receiving operators see warmup traffic clustering legitimate domains with spam (Jarland Donnell, MXroute, mailop msg21660), and Resend's own guidance says artificial engagement "can eventually damage your reputation".

**Complaint arithmetic (illustrative sensitivity, not a measured rate — real denominators are per-provider/per-day):** at 500 recipients, one spam-button press = 0.2% — 2.5x Resend's 0.08% account line; two = 0.4% — past Gmail's 0.30% never-exceed, after which mitigation eligibility requires 7 consecutive days below 0.3% (support.google.com/a/answer/81126, /14229414). SES enforcement corroborates the shape: review at 0.1% complaint / 5% bounce, possible pause at 0.5% / 10% (faqs-enforcement.html). At small N the breaker treats **one** complaint as an emergency stop.

## 5. Framework conformance

**Codex ran the built probe: exit 1, eight failure classes. This is the observed P0 list** (wave-1's "at least 7, inferred" is superseded):

1. Command-object manifest shape (agent_info.rs:9-60 — free-text keys → `{description,args[],options[],aliases?}`)
2. `global_flags` object with `--json` AND `--quiet`
3. Config metadata nested as `config.{path, env_prefix}`
4. Piped `--help` success envelope
5. `Tips:` / `Examples:` sections in help
6. Routability (`$BIN <cmd> --help` exits 0 for every advertised name)
7. Unknown-command exit code — **observed 2**, must be 3 (`Cli::parse()` owns the failure, main.rs:25; the clap-exit-2 inference is now observation)
8. Unknown-command JSON error envelope on stderr

One entry-point rewrite (`try_parse`, pre-scan `--json`, intercept DisplayHelp/DisplayVersion, own every exit) plus the manifest restructure clears most of these. Add the **pinned probe to CI and release** (ci.yml:36, release.yml:70).

**Prose-requirement gaps (not probe failures — don't conflate), with Codex corrections folded:**

- **Doctor**: keep `health` exit-0 (tests/cli.rs:299 requires it; a visible alias cannot carry different exit semantics) and add a **separate `doctor`** over a shared check collector with spec exit-2-on-fail. Boris sign-off still required for the dual-command surface.
- **Config**: the real missing contract is **defaults → TOML → environment layering** — loading is defaults-or-TOML today (config.rs:138). Correct the wave-1 secret claim: config stores secret env-var *names*, not resolved secrets; `config show` + `mask_secret()` still land together, but the leak risk was overstated.
- **Update**: ship `update --check` / instructions-only first — the command is a stub (update.rs:5) and releases produce source/Homebrew artifacts, not platform binaries + checksums (release.yml:133), so atomic self-replace has nothing to download yet.
- **Aliases**: `template inspect` already has `info` (cli.rs:382) — don't re-add; do NOT blanket-map `delete` onto semantically different GDPR erase operations.
- **Lock**: keep the SQLite CAS lock (stronger than the spec's lock-file); fix cleanup/heartbeat/ownership (§2 e). Renaming `--force-unlock` to the vaguer `--force` is not the compliance issue and increases agent risk — keep the specific flag; fix the lock-held exit code 1 → 3.
- **Tests**: rewrite assertions that require string-valued manifest commands (tests/cli.rs:35).
- `partial_success`/`all_failed` envelope statuses ride in with the outcome type (§2 d / §3 step 2), not as a bolt-on.

## 6. Open questions

**Resolved since wave 1** (for the record): unsubscribe-presence gate unnecessary — rendering is unconditional per-recipient (pipeline.rs:396, :473) and lint requires the body placeholder (template/render.rs:166); `AppError::RateLimited` already in use (email_cli.rs:472); clap parse-error exit observed = 2; Gmail's November-2025 enforcement ramp is now confirmed by Google's own FAQ (closes wave-1 Q8); PMT v1 status settled by arbitration (alive, deprecation postponed — gate on v2 Spam Rate + Compliance); Gmail Manage Subscriptions is a **July 2025** feature (wave-1 said 2026).

**Open — carried:**
1. Multi-domain ledger scoping: the ledger is per-identity only under the single-configured-sender assumption. Does anyone run multiple sending domains from one DB? (Boris.)
2. DMARC lookup: shell out to `dig` (graceful `unknown`) vs a resolver crate. (Boris.)
3. Postmaster data floor: does PMT show anything at ~500/day? Operator consensus says ~100/day to Gmail for data; Google publishes only "too low… privacy". Enroll and observe.
4. Resend support: per-provider deferral detail, and whether Broadcast vs transactional traffic ride separate IP pools (undocumented — ask before finalizing per-provider monitoring).

**Open — new (Codex):**
5. Multi-machine ledger: a second machine/DB is invisible to any local guard. Central domain-budget service, or an OS/capability boundary where the agent never holds the Resend key?
6. Out-of-band override token: short-lived approval bound to identity + recipient cap + broadcast hash + expiry. Who mints it, where is it verified, how does a human issue it without the agent being able to?
7. Tracking-domain lifecycle: `links.<domain>` carries its own warmable/burnable reputation (Atkins, mailop msg21664; Iverson's Bitly cases); rewritten URLs can look suspicious (resend.com/docs/dashboard/emails/deliverability-insights); the old tracking CNAME must stay alive after any migration or already-sent links break (docs/dashboard/domains/tracking). Age-gate tracking subdomains before first bulk use (Spamhaus ZRD blocklists all first-24h domains; practitioner aging bands 30–90 days — Atkins, Vernhout)?
8. Cache-file retention: batch files contain recipient addresses, personalized HTML, and live unsubscribe tokens in persistent cache (pipeline.rs:481). Permissions, deletion policy, incident retention?
9. Provider-level budgeting: a total-identity cap can still send the entire first cohort to Gmail. Per-MX sub-budgets in the governor?
10. Clock integrity: the ledger trusts local wall time; pacing needs monotonic timing + validated UTC.
11. Seed-panel volume must count against provider and ramp budgets.
12. Consent/legal basis: "personally known" is not newsletter consent — does the CLI grow import provenance/consent metadata (also the anti-list-bombing COI pipeline for `web/`, M3AAWG)?
